1. ゲームエンジンとレンダリング

DirectX 12、Metal、Vulkan:
- DirectX 11とは完全に異なるAPI。

フレームバッファ:
- 画面を描画するための機能。

GPUチップ内部メモリの重要性:
- 内部メモリのセットアップには時間がかかる（ステート変更、テクスチャ変更時）。
- 内部キャッシュを一度クリアして再度埋める必要がある。

ラスタライザ:
- 頂点をピクセル化する工程。
- 三角形ポリゴンをピクセルに変換する。

アンチエイリアシング:
- ピクセル補正（スムージング）。

テクスチャを大きなアトラスにまとめる理由、頻繁なテクスチャ切り替えがよくない理由:
- GPUキャッシュではタイル状/ジグザグ状に格納されることが多い。
- メモリアドレスの連続性が悪いと処理が遅くなる。

Zバッファリング:
- ピクセル単位で深度値`z`を比較する。
- depth bufferを使用する。
- 面同士が近すぎると`z-fighting`が起きる。

Chris Hecker - Texture Mapping:
- 双曲線補間。
- テクスチャマッピング。

レンダリング方式の種類:
- Forward Rendering
  - 最も古い基本方式。
  - 手前のものからその都度描く。
  - VRの再普及で再びよく使われる。
  - オブジェクトごとの処理（頂点、シェーディングなど）後に描くため、オーバードローが多くなり無駄が大きい。

- Deferred Rendering
  - 遅延レンダリング。
  - 複雑なライティング、影、反射に向く。
  - 必要な情報をバッファに貯め、最後に可視部分だけ計算する。
  - 最終的に見えるピクセルだけを計算するので高速。
  - G-buffer（Geometric Buffer）を使う。
  - 欠点: 透過マテリアルは別処理が必要。
  - 4K以上など高解像度ではG-bufferのメモリコストが大きく、モバイルには不向き。

- Forward+ Rendering
  - Light HeatMapやクラスタ化されたライト情報を使う。
  - どこでライティング計算が必要かを事前に計算する。
  - 比較的新しい実用技術。

- TBDR: Tile Based Deferred Rendering
  - ハードウェアレベルで動くため、モバイル端末側で実行され、直接制御は難しい。
  - モバイルで主に使われる。
  - タイル単位で描画し、メモリ消費を抑える。

エンジン別レンダリングパイプライン:

Unity

URP
- Forward、Deferred、Forward+を提供。

HDRP
- 高品質リアルタイムレンダリング向け設計。

Unreal Engine 5
- 基本はDeferred。
- 必要ならForwardを有効化できる。

VRは左右2枚を描画する必要があるため、
一般的にはDeferredよりForwardが好まれる。

Godot Engine
- Forward+
- モバイルではタイルベース
- GLESベースレンダリングと互換

チャプター2 レンダリングパイプラインの構造

* 座標系の変換

Transform Pipeline（OpenGL）:
- 最も基礎となる骨格。

座標系の種類:
- 左手系（DirectX）と右手系（OpenGL）。
- 軸の回転方向の基準が左右で逆。
- このノート基準では、右手系は時計回りが`+`、左手系は反時計回りが`+`。
- 外積の結果方向は座標系によって逆になる。
- 法線ベクトルの向きも座標系で反転する。
- 数式だけでは区別できない。外積公式自体は同じ。

ほかに使う座標系/概念:
- 球座標系（宇宙的な視点）
- BRDF（双方向反射分布関数）: 入射光と反射光など
- 球面調和関数（量子力学、ライティング）
- 円筒座標系

テクスチャ座標系（UV原点と向き）:
- 左下原点のケースと左上原点のケースがある。
- テクスチャが上下反転して見える理由の一つ。
- Unityは左下原点。

行列変換の数学的特性:
- 2D変換なのに、なぜ`3x3`で表すのか。
- 平行移動: 加算
- 拡大縮小/回転: 乗算
- すべてを乗算で統一する枠組みが必要になる。

行列演算の混在:
- 回転は乗算、移動は加算。
- すべて乗算で表せるようにしたものがアフィン変換。
- そのために座標を1つ追加する。

同次座標系への拡張:
- 2D変換をすべて`3x3`行列の積で表せる。
- 移動、回転、拡大縮小、せん断を適用可能。

Rigid / Euclidean 変換:
- 長さと角度を保存。
- 恒等、移動、回転。

剛体:
- 形状は変わらず、位置と回転だけが変わる。

回転変換行列:
- 直交行列。
- 逆行列、転置行列、角度の符号反転は純回転で密接に対応する。

相似変換（スケーリング）:
- 等方スケーリング。
- 角度は保存される。

線形変換:
- 定数倍。
- 行列の積で表現可能。

Affine Transformation:
- 平行線が保存される。
- 代数的には線形変換式に定数項を追加した形。
x' = ax + by + c
y' = dx + ey + f
- アフィン変換全体は1次多項式として表せる。
- Orthographic（正射影）に関連。

Projective Transformation:
- 直線は保たれるが、平行線は保たれない。
- 遠近（Perspective）投影。
- アフィン変換だけでは表現しきれない。

一般的な非線形変換:
- 直線が保存されない。
- ピンクッション歪み、バレル歪み。
- VRでも使用（例: Oculus Riftのバレル歪み補正）。昔は実レンズ問題だったが、今はレンダリング効果で補正することが多い。

魚眼レンズ変換:
- 水中表現などに活用できる。

360度パノラマ変換などもある。

変換順序の重要性:

同じ90度回転と同じx方向移動でも結果は変わる。
1. 90度回転 -> x方向移動
2. x方向移動 -> 90度回転

変換の合成と行列積:
- 行列積は可換ではない。
- 合成順序で結果が変わる。

変換順序と行列積順序:
- D3Dでは変換順序と行列積順序が一致（位置ベクトル: 行ベクトル）。
- OGLでは変換順序と行列積順序が逆（位置ベクトル: 列ベクトル）。

これは行優先/列優先の取り扱い差による。

ワールド変換の順序:
- 標準的には
- SRT: Scale -> Rotation -> Translate

Unityのオイラー角回転順序: `Z, X, Y`

X軸回転: Pitch
Y軸回転: Yaw
Z軸回転: Roll

この順序はカメラのジンバルロック回避に有利。

モデル空間 -> ワールド空間

回転の表現:
- 2Dでは複素数で表現可能。
- 3Dではクォータニオンを使う。

カメラ変換行列:
- ビュー変換。
- カメラ変換は、ワールド空間でのカメラ剛体変換の逆変換。
- `(R * T)^-1 = T^-1 * R^-1`

法線ベクトルの変換特性:
- 座標点とは異なる取り扱いが必要。
- 等方スケーリングなら問題が少ないが、片軸だけスケールすると法線が変わる。
- 異方スケーリングでも法線は正しい垂直方向を示す必要がある。
- 2ベクトルの内積が`0`なら垂直。

透視投影変換行列:
- パイプライン変換で最も難しい部分の一つ。
- 深度値の再マッピングが難所。
- `z`値はz-bufferingに適した範囲が必要。
- FOVや視野範囲を調整しやすい必要がある。

z-bufferingと透視投影を適用した後:
- 座標はNDC（Normalized Device Coordinates）になる。
- ビュー錐台（Frustum）をNDCへ変換する必要がある。

チャプター3 レイトレーシング実習

GLSL

光の性質:
- 光の二重性（粒子と波動）。

スネルの法則:
- 屈折率から屈折角を求められる。

屈折率:
- 真空 1.0、空気 1.003、水 1.33...

波動の屈折:
- ホイヘンスの原理。

光エネルギー:
- 反射、吸収、屈折（エネルギー保存）。

最近の物理ベースエンジンでは光エネルギーの挙動も考慮する:
- 光が入射すると一部は反射し、一部は吸収される。

反射色の原理:
- マテリアル特性と固有色（アルベド）。

反射特性: フレネル効果
- 入射角が浅いほど反射が強い。
- 入射角によって反射のされ方が変わる。
- フレネル方程式は偏光現象を扱う。
- 横波/縦波成分で反射率が異なる。
- エネルギー保存の扱いにもフレネル項が関わる。
- 誘電体と金属ではフレネル反射率が異なる。
- 屈折時は波長ごとに屈折率が変わり、虹になる。
- 波の干渉（強め合い・打ち消し）もある。
- 例: 石けん膜干渉（強め合い）、メガネの反射防止コート（打ち消し）。
- 波の回折。
- 光散乱（Scattering） -> 雲、レイリー散乱、ミー散乱。
- 青空は大気中のレイリー散乱による。

- 「Physics and Math of Shading」で詳しく学べる。

Phong Reflection Model Formula = Specular + Diffuse + Ambient
- 厳密なエネルギー保存は無視し、値を足し合わせて表現する。
- 旧世代CGの典型（プラスチック感のある見た目）。

Specular:
- 正反射、鏡面反射、滑らかな表面。

Diffuse:
- 拡散反射、ざらついた微細表面。

- Face normal、Vertex normal、Pixel normal（最新）。
- Phong shading: 頂点法線を補間し、ピクセルごとにライティング計算を行う。

Normal map:
- 欠点: フェイク手法なのでジオメトリ自体は変化しない。近くで見ると破綻が分かる。

最近は:
- PBRのような、より写実的な反射モデルを使う。
- 特に人体の肌表現で重要。
- 金属ではBRDFでより現実的な材質表面を表現できる。

PBR: Physically Based Rendering

GI: Global Illumination
- 間接光を扱う。
- 古典的手法 -> ベイクして事前計算。
- 最近はUnreal Engine 5のLumenのようにリアルタイムGIも提供される。
- Ray tracing: 反射・屈折。
- Radiosity: 拡散反射中心。

フラクタル表現:
- マンデルブロ集合の式を使う。
- 複素数空間で表現する。

レイトレーシング:
- 入射角と反射角が等しい法則を使う。
- カメラからレイを飛ばす方法は、カメラに届く光路を逆にたどる考え方と等価。
- 比較的シンプルで小規模なレイ構成でも実装可能。
- 影の描画がしやすい。
- ハードウェアレイトレーシングが可能になって以降、
- 既存はポリゴンレンダリング、影や反射のみレイトレーシングというハイブリッド構成が多い。
- レイトレーシングでは球が特に扱いやすいプリミティブ。
- 球との交差判定は2次方程式で比較的簡単に解ける。
- 結論: 数学をしっかりやること。
