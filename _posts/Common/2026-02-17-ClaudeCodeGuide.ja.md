## Claude Codeとは？

**Claude Code**はAnthropicが開発した**Agentic Coding Tool**です。ターミナル（CLI）で直接実行され、自然言語でコードを書く、リファクタリング、デバッグ、git管理など、ほとんどすべての開発作業を実行できます。

既存のAIコーディングアシスタント（Copilot、Cursorなど）との最大の違いは、**プロジェクト全体をコンテキストとして理解**し、ファイルシステムを直接読み書きできる**エージェント（Agent）**という点です。

＃＃II。初期設定

### 1. ログイン

インストール後、端末からログインを続行します。

```bash
claude login
```

ブラウザが開いたら、Anthropicアカウントでログインし、CLI toolタイプを選択します。

### 2. トークン使用量の確認

[https://claude.ai/settings/usage](https://claude.ai/settings/usage) で現在のトークン使用量を確認できます。

> **トークン（Token）とは** LLMがテキストを処理する最小単位です。英語基準約1単語=1.3トークン、韓国語は1文字=約2～3トークンと計算されます。 Claude Codeは**入力トークン（プロンプト+コンテキスト）**と**出力トークン（応答）**の両方を消費します。
{：.prompt-info}

＃＃III。 IDE連動(おすすめ環境)

ターミナルですぐに使えますが、**IDEに接続して書くことを強くお勧め**します。コードの変更をリアルタイムで確認でき、はるかに効率的です。

### VS Code (おすすめ)

1. [VS Code](https://code.visualstudio.com/)のインストール
2. **Open Folder**でプロジェクトフォルダを開く
3. トップメニューで **Terminal > New Terminal** を開く
4. 端末に `claude` を入力して実行

```bash
# プロジェクトディレクトリで実行
cd /path/to/your/project
claude
```

> Claude Codeは、実行時にプロジェクトルートの`CLAUDE.md`ファイルを自動的に読み込み、プロジェクトのアーキテクチャ、コーディングルール、および作業手順を把握します。その後、要件に合わせてより正確で関連性の高い回答を提供します。
{：.prompt-tip}

### JetBrains (Rider、IntelliJなど)

JetBrains IDEの組み込み端末でも同様に `claude` 命令で実行可能です。ただし、個人的にはVS Codeの端末の使い勝手が良かったです。

## V.最新機能（v2.1.x、2026年2月現在）

Claude Code v2.1.xで追加された主な機能です。

### 1. `/insights` - 使用パターン分析レポート

`/insights`は、過去30日間のセッションデータを分析して**インタラクティブHTMLレポート**を生成する命令です。

```bash
# Claude Code セッション内で実行
/insights
```

**分析対象：**
- `~/.claude/`に保存されたセッション履歴（プロンプト+ Claude応答）
- ツール使用パターン、エラーパターン、インタラクションパターン
- ソースコード自体は分析しない（プライバシー保証）

**レポートに含まれる内容：**|アイテム|説明
|:---|:---|
| **使用統計** |メッセージ数、セッション数、ファイル修正履歴|
| **プロジェクト別分析** |どのプロジェクトでどんな仕事を主にしたのか
| **ツール使用パターン** |どのツールを最も使用したかをグラフで視覚化
| **言語別使用量** |どのプログラミング言語で作業したのか
| **ボトルネックポイント** |生産性を低下させるパターンの識別
| **改善提案** | CLAUDE.mdにコピー可能なカスタム提案、カスタムスキル推奨、フック推奨|

レポートは `~/.claude/usage-data/report.html` に保存され、自動的にブラウザで開きます。

> `/insights`の提案には**Copyボタン**が含まれており、すぐにCLAUDE.mdに貼り付けることができます。独自の使用パターンに基づいたカスタム最適化提案なので非常に便利です。
{：.prompt-tip}

### 2. Extended Thinking & Effortシステム

Claude Code は **Extended Thinking(拡張推論)** 機能をサポートします。これはClaudeが応答の前に**内部的に深く思考**する過程を経ることで、複雑な問題で精度が大幅に向上します。

**Thinkingトグル方法：**

|方法説明
|:---|:---|
| `Tab`キー|セッション内で即座にシンクモードオン/オフを切り替えます。
| `/config` |設定でシンキングを有効/無効にする
|環境変数`MAX_THINKING_TOKENS=8000` | Thinkingトークン予算を直接指定する（最小1,024）

**Effortレベル（Opus 4.6のみ）：**

`/model`コマンドでモデルを選択するとき**Effortレベル**を調整できます。これはClaudeがどれだけ深く推論するかを制御します。

| Effort |用途|トークン消費|推奨状況|
|:---:|:---|:---:|:---|
| **Low** |クイックレスポンス、シンプルな操作|少ない簡単な分類、クイック検索、一括処理|
| **Medium** |バランスのとれたパフォーマンス通常|一般的なコーディングタスク
| **High**（デフォルト）|最高品質の推論|たくさん|複雑な推論、難しいコーディング問題
| **Max** |絶対最高性能|最大非常に複雑なアーキテクチャ分析、多段階推論

```bash
# Effort レベル環境変数に設定
CLAUDE_CODE_EFFORT_LEVEL=high claude

# またはセッション内で /modelに変更
/model
# → モデル選択後 Effort スライダーの調整
```

> **Adaptive Thinking**：Opus 4.6では、<<CODE_BLOCK_65>>>を直接指定する代わりに、**Adaptive Thinking**をお勧めします。 Claudeは質問の複雑さに応じて**動的**シンクの深さを調整します。 Effortレベルでこの範囲を制御できます。
{：.prompt-info}

### 3. Fastモード

**高速モード**は同じモデルを使用しますが、**出力トークンの生成速度を最大2.5倍に高める**機能です。モデルが変わるのではなく、同じモデルの高速推論パスを使用します。

```bash
# セッション内でトグル
/fast

# 有効になると雷アイコン(↯) 表示
# 設定はセッション間で維持されます
```

> Fastモードは**プレミアム料金**($30/$150 per MTok)が適用されます。迅速な反復を必要とするプロトタイピング段階では便利ですが、コストを考慮してオプションで使用してください。
{：.prompt-warning}

### 4. Agent Teams (研究プレビュー)

**Agent Teams**は、複数のClaude Codeインスタンスが**チームとしてコラボレーション**する機能です。 1つのオーケストレーターがタスク全体を調整し、各サブエージェントが異なる部分を並列に処理します。

```
┌─────────────────────────────┐
│      Orchestrator (リーダー)      │
│    全体の作業調整と配布       │
└──────┬──────┬──────┬────────┘
       │      │      │
  ┌────▼──┐ ┌─▼───┐ ┌▼─────┐
  │Agent 1│ │Agent2│ │Agent3│
  │バックエンド  │ │フロント│ │テスト │
  └───────┘ └─────┘ └──────┘
```

- 各サブエージェントは独自の**tmux pane**で実行
- 独立したタスクを**並列に処理**することで大規模な作業速度を向上
- 現在の**研究プレビュー**ステップ（変更することができます）

### 5. Auto Memory (自動メモリ)

Claude Codeが**自動的にセッション間のコンテキストを記憶**する機能です。手動設定なしでバックグラウンドで動作します。

**動作方法：**

|表示|意味|
|:---|:---|
| `Recalled X memories` |セッションの開始時に前のセッションの記憶をロードする
| `Wrote X memories` |セッション中に現在のジョブのスナップショットを保存します。

- プロジェクトパターン、主な命令、ユーザーの好みを自動保存
- `~/.claude/projects/<project-hash>/<session-id>/session-memory/summary.md`に保存
- `#`キーでCLAUDE.mdにすばやくメモリを追加可能

> Auto MemoryはCLAUDE.mdとは別のものです。 CLAUDE.md はユーザーが直接管理する明示的なルールであり、Auto Memory は Claude が自動的に抽出する暗黙のコンテキストです。 2つを一緒に活用すると最も効果的です。

## VI.コンテキストウィンドウの深い理解

Claude Codeを効果的に使用するには、「Context Window」の理解が不可欠です。

### コンテキストウィンドウとは？

コンテキストウィンドウは、LLMが**一度に処理できるトークンの最大サイズ**です。 Claudeモデルに基づいて：

|モデル|コンテキストウィンドウ|特徴|
|:---|:---|:---|
| **Claude Opus 4.6** | 200K（1Mベータ）|最も強力な推論、複雑なアーキテクチャ分析に適しています。
| **Claude Sonnet 4.5** | 200Kトークン|高速応答、一般的なコーディング作業に効率的

> **200Kトークンはおよそどの程度？** 英語基準約15万語、A4約300ページ分量です。しかし、コードと韓国語はトークン効率が低く、実際には少ないです。 1Mトークンは、その約5倍のA4 1,500ページ分量です。
{：.prompt-info}

### コンテキストウィンドウが重要な理由

LLM はリクエストごとに **Stateless** です。つまり、Claude Codeとの新しいチャットは、**毎回新しいチームメンバーとのコラボレーション**と同じです。前の会話の内容は、コンテキストウィンドウ内になければ記憶されません。

```
[システムプロンプト] + [CLAUDE.md] + [会話履歴] + [現在の質問] = トークンの総使用量
                                                              ↕
                                                    コンテキストウィンドウの制限
```

コンテキストウィンドウがいっぱいになると：
1. **Auto-compact**が発動→前の会話が自動要約
2. 要約過程で**情報損失/歪み**発生
3. 既存計画の文脈が消えることがある

### Auto-Compactの構造的問題

自動コンパクトを実行すると**元に戻すことはできません**要約すると、既存の会話は永久に失われ、次の問題が発生します。

|問題説明
|:---|:---|
| **コンテキスト歪み** |要約過程で、オリジナルの意味が変質する可能性があります。
| **時間的コンテキスト損失** | 「前に話したAを元にBをしよう」などの逐次的文脈が消える
| **情報の破片化** |関連情報が分離されて推論品質が低下する

> Auto-compactは会話の「中点」を要約する傾向があり、Phase 1全体を要約するのではなく、Phase 1.2～1.5付近を要約してしまう現象が発生します。
{：.prompt-warning}

### コンテキスト管理戦略

**1。定期的な手動 `/compact`**
- Auto-compactに頼らず、適切なタイミングで<<CODE_BLOCK_73>>>を直接実行
- `/compact 現在 Todo リストと進捗状況を中心に`のようにコアコンテンツを指定できます

**2。タスク単位で `/clear`**
- 1つの大きなタスクを複数のセッションに分ける
- 各セッションは `/clear` でクリーンなコンテキストで開始

**3。 CLAUDE.mdの活用**
- 毎回繰り返す必要があるプロジェクト情報をCLAUDE.mdに記録する
- 新しいセッションごとに自動的にロードされるので、コンテキストを惜しまない**4。外部文書を参照**
- 複雑な計画は別の `.md` ファイルで作成し、 `@ファイル名` として参照
- コンテキストウィンドウ内で計画を直接維持するよりも効率的

＃＃VII。 CLAUDE.mdファイルの理解

### CLAUDE.mdとは？

CLAUDE.mdはプロジェクトの**「AIオンボーディングドキュメント」**です。新しいチームメンバーが入社すると、プロジェクトウィキを読むように、Claude Codeはセッションごとにこのファイルを読み、プロジェクトを理解します。

### CLAUDE.mdに含めるべき内容

```markdown
# CLAUDE.md

## Project Overview
- プロジェクトの説明, 技術スタック, アーキテクチャの概要

## Build & Development Commands
- ビルド, テスト, 実行命令

## Code Conventions
- コーディングスタイル, ネーミングルール, ファイル構造

## Architecture
- 主なディレクトリ構造, コアモジュールの説明

## Common Patterns
- プロジェクトでよく使われるパターン
- "これをしないでください" ルール (Claudeが繰り返す間違い防止)
```

### メモリーネスティング (Memory Nesting)

CLAUDE.mdは[複数階層で構成]（https://code.claude.com/docs/en/memory#how-claude-looks-up-memories）できます。

|場所範囲用途|
|:---|:---|:---|
| `~/.claude/CLAUDE.md` | **グローバル（ユーザー単位）** |すべてのプロジェクトに共通する規則
| `./CLAUDE.md` | **プロジェクトルート** |そのプロジェクト全体に適用|
| `./src/CLAUDE.md` | **サブディレクトリ** |特定のモジュール/ディレクトリにのみ適用|

> `/init`コマンドで自動生成されたCLAUDE.mdを**そのまま使用しないでください**プロジェクトのコードルール、設計方向、頻繁な間違いを直接追加し、継続的に改善する必要があります。
{：.prompt-warning}

＃＃VIII。計画 - 実行ワークフロー (Plan & Execute)

###コア原則

Claude Codeの活用の鍵は、「計画」と「実行」の段階を明確に分離することです。

###モデル選択戦略

|ステップ|おすすめモデル|理由
|:---|:---|:---|
| **計画策定** | Opus |複雑な推論、アーキテクチャ分析、戦略策定に強い
| **コードの実装** | Sonnet |迅速な対応、費用対効果、シンプルな実装作業に最適

```
1. /model → Opus 選択
2. Shift+Tab 2回 → Plan モードに入る
3. 計画の策定と Todo リストの作成
4. /model → Sonnetに変更
5. 段階的なコード実装の進行
```

### Planモードベースのワークフロー

**ステップ1：ナビゲーションと計画の確立（Explore & Plan）**

Planモード（`Shift+Tab` 2番）で起動します。

- 実施戦略の策定
- タスクを**独立してテスト可能なステップ**に分解
- 推定所要時間の見積もり
- UI/ライブラリに関する事項を含む

>詳細なプロンプトが重要です。 「この機能を実装するための作業をステップごとに分けてください。各ステップは独立してテスト可能でなければならず、予想所要時間も推定してください」のように具体的に指示します。
{：.prompt-tip}

**ステップ2：計画の確認と実行**

計画が満足している場合：
1. `Shift+Tab`へ**Auto-accept editsモード**を切り替える
2. Claudeが計画に従って**一度に(1-shot)実装**進行

**Planモードの利点：**

|利点|説明
|:---|:---|
| **コンテキスト継続** | PlanモードのTodoリストはセッションが長くなっても維持され、ジョブの連続性が保証されます。 |
| **動的計画の修正** |実装中に欠けている部分があれば、計画をすぐに修正できます。 |
| **ハルシネーションの減少** |明確な計画があれば、Claudeが奇妙な方向に行くのを防ぎます。 |

>可能であれば、<<CODE_BLOCK_85>>>の形式で現在作業しているタスクのチェックリスト文書を作成し、継続的に更新するように指示することをお勧めします。
{：.prompt-tip}

### PRD（Product Requirements Document）の活用

単純にPlanモードでTodoリストを作成するのも良いですが、**大規模な作業やリファクタリング**ではPRDを活用することが効果的です。

PRDには以下が含まれます：
- **目標と背景**：なぜこれが必要なのか
- **要件**：機能的/非機能的要件
- **範囲**: 含める/含まないもの
- **設計方向**: アーキテクチャの決定事項
- **成功基準**：完了条件

＃＃IX。安全性と制御(Safety & Control)

###権限モードについて

|モード|説明使用時期
|:---|:---|:---|
| **基本モード** |ファイル修正/コマンド実行時に毎回承認が必要重要なプロジェクト、学習フェーズ|
| **Auto-accept** |ファイルの変更は自動承認されます。計画が確定した後の実施段階
| **dangerously-skip-permissions** |全権限自動承認サンドボックス環境でのみ使用することをお勧めします。

### `/permissions`による細かい制御`--dangerously-skip-permissions`の代わりに`/permissions`として頻繁に使用される安全なコマンドのみを事前に許可することをお勧めします。

```bash
# .claude/settings.jsonに保存され、チームと共有可能
# はい: git, npm, build コマンドは事前許可
```

###安全な使用原則

1. **最初は承認モード**で一つずつ確認しながら進行
2. 計画が確定したら Auto-accept に切り替える
3. 機能の実装が予想と異なる場合は `Esc 2回`(Rewind) に戻す
4. 計画自体が間違っている場合は、計画を修正

## X. 13の高度な活用のヒント（Claude Codeの創設者のワークフロー）

Claude Code開発者が直接共有した本番のヒントです。

### 1. 並列実行環境の構成

ターミナルで**5個のClaudeを並列実行**します。タブに1〜5番号を付けて、[システム通知]（https://code.claude.com/docs/en/terminal-config#iterm-2-system-notifications）で入力が必要な時点を把握します。

### 2. Webとローカルの並列操作

- `claude.ai/code` ウェブでも**5～10個のClaudeを追加で並列実行**
- ローカルセッションをウェブにハンドオフ（`&`使用）するか、`-teleport`に双方向に切り替える
- iOSアプリでセッションを開始して後で確認する方法も活用

### 3. モデルの選択: Opus with thinking

- すべてのタスクに**Opus 4 with thinking**を使用する
- Sonnetより大きくて遅いが、**調整(steering)が少なく必要**と**ツール活用能力に優れる**
- 結果として、小型モデルよりも**ほぼ常に速い**最終結果の導出

### 4. CLAUDE.mdによるチーム単位の知識の蓄積

- チーム全体が共有する**シングルCLAUDE.mdファイル**を維持
- gitにチェックインし、チーム全体が**週単位で何度も貢献**
- Claudeが間違った行動をするたびに追加**次のような間違いを防止**

### 5. コードレビュー時にCLAUDE.mdを更新

- コードレビュー時に同僚PRに**@.claudeをタグ**してCLAUDE.mdに内容を追加
- **Claude Code GitHub Action**(`/install-github-action`)を活用

### 6. Planモードと自動受付ワークフロー

- ほとんどのセッションを** Planモード**（`Shift+Tab` 2回）で開始
- 計画が満足するまで繰り返し協議
- 確定後**Auto-acceptモード**に切り替えると**一度に(1-shot)完成**
- **良い計画が本当に重要**

### 7. スラッシュコマンドによる繰り返し操作の自動化

- 頻繁に実行する「inner loop」ワークフローごとにスラッシュコマンドを使用
- `.claude/commands/`ディレクトリに保存され、gitにチェックイン
- 例: `/commit-push-pr` スラッシュコマンドを毎日数十回使用
- [インラインbash]（https://code.claude.com/docs/en/slash-commands#bash-command-execution）でgit statusなどをあらかじめ計算して不要なモデル往復防止

### 8. サブエージェントの活用

複数の[サブエージェント]（https://code.claude.com/docs/en/sub-agents）を定期的に使用する：
- **code-simplifier**: 操作完了後のコードの簡素化
- **verify-app**: エンドツーエンドのテストのための詳細なガイドライン

### 9. PostToolUse フックでコードをフォーマットする

- **PostToolUseフック**を使用したコードフォーマットの自動処理
- Claudeがデフォルトでうまくフォーマットされたコードを生成し、フックが**残り10%を処理**

### 10. 権限管理方法

- `--dangerously-skip-permissions`未使用
- 代わりに`/permissions`で安全なbashコマンドを**事前許可**
- `.claude/settings.json`にチェックインしてチームと共有

### 11. ツール統合の活用

Claude Codeが**外部ツールを代わりに使用**:
- **Slack**の検索と公開（MCPサーバーを活用）
- **BigQuery**クエリの実行（bq CLI）
- **Sentry**からエラーログを取得する
- MCP設定は<<CODE_BLOCK_100>>>にチェックインしてチームと共有

### 12. 長時間の作業処理方法非常に長い作業の場合は、3つの方法を選択してください。
- **(a)** 完了時に **バックグラウンドエージェント**でジョブを確認するようプロンプト
- **(b)** [エージェント Stop Hook] (https://code.claude.com/docs/en/hooks-guide) を使用してより決定論的に検証
- **(c)** [ralph-wiggumプラグイン](https://github.com/anthropics/claude-plugins-official/tree/main/plugins/ralph-wiggum)を使用する

### 13.最も重要なヒント：検証フィードバックループ

> Claude Codeで優れた結果を得るための**最も重要な要素**：Claudeに**作業を検証する方法を提供**することです。
{：.prompt-warning}

このフィードバックループがあると、最終結果の品質が**2〜3倍向上**されます。
- bash コマンドの実行による結果の確認
- テストスイートの実行
- ブラウザやシミュレータでアプリをテストする
- [Chrome拡張機能]（https://code.claude.com/docs/en/chrome）でUIをテストする

**検証プロセスをしっかりと構築するために投資することが最も価値のあるものです。**

＃＃XI。 SDD(Spec-Driven Development)

### SDDとは？

SDD（Spec-Driven Development）は「Claudeに言葉をつける開発」から抜け出し、**チームが共有する技術仕様を基準にAIが実装/テストまで行う開発方式**です。

**SDDの重要な価値：**
- コンテキスト損失の最小化
- ハルシネーションの減少
- チームコラボレーションの効率化

### Spec Kitのインストールと使用

[Spec Kit]（https://github.com/github/spec-kit）はSDDを実現するためのツールです。

```bash
# Spec Kit 初期化
specify init --here
# → 'y' 入力
# → 'claude' 選択
# → 'sh' 選択
```

インストール後に新しいセッションでclaudeを実行すると、新しい命令が追加されます。

### Spec Kitワークフロー

|命令|ステップ|説明
|:---|:---|:---|
| `/speckit.constitution` |規約設定| CLAUDE.mdとプロジェクトコードルール、アーキテクチャベースの規約
| `/speckit.specify` |要件定義| **What/Why** ベース。技術スタックは書きません。
| `/speckit.clarify` |詳細不足技術書の補完
| `/speckit.plan` |計画の策定技術スタック/アーキテクチャをこの段階で決定する
| `/speckit.tasks` |タスクの分解実現可能な単位に分解
| `/speckit.implement` |実装|実際のコードを書く|
| `/speckit.analyze` |分析（オプション）|文書間の一貫性チェック|
| `/speckit.checklist` |チェックリスト（オプション）|品質チェック|

### `/speckit.specify`作成ガイド

`specify`は、**何を作成するか（What /Why）**を書き込むステップです。実装方法（How）は`/speckit.plan`でカバーされています。

**WHAT（要件）**
- ユーザーができること（行動）
- システムが保証しなければならない規則（検証/制約/互換性）
- 出力(アセット、プレハブ、登録、テスト環境)
- 成功基準(時間制約、レガシー互換など)

**WHY(必要性)**
- 既存の問題（ボトルネック、重複、不一致、コンテキストの損失）
- 成功指標への貢献度

**作成禁止（これは`/plan`>）**
- ~~"BaseWeaponにDIコンテナを入れて..."~~
- ~~"BulletRecipeをScriptableObjectにして..."~~
- ~~"ECSで最適化して..."~~

## XII.プラグインの活用

### プラグインのインストール方法

```bash
# Claude Code 実行後
claude

# マーケットプレイスからプラグインをダウンロード
/plugin marketplace add wshobson/agents

# 必要なプラグインオプションのインストール
/plugin install game-development
/plugin install debugging-toolkit
/plugin install code-refactoring
```

[wshobson/agents]（https://github.com/wshobson/agents/tree/main）リポジトリには、オーケストレーションによって最適化されたさまざまなエージェントが含まれています。

### プラグインの仕組み

インストールされたプラグインは**自動的に状況に合わせて利用**されます:

- **自動活用**: 作業特性に合うと自動的にそのエージェントが動作
  - コード作成後→code-reviewerエージェント
  - エラー発生時 → debugger エージェント
- **明示的なリクエスト**：「コードレビューしてください」→code-reviewerを有効にする
- **状況判断**: 単純作業には別途エージェントなしで直接処理

## XIII.注意事項

### Unityインターロック制限現在、Claude CodeとUnity Editorの間に直接接続する手段はありません。これにより、Unityエディタのエラーコードが正しく認識されない場合があります。ただし、**ロジックエラー分析やコード意図把握性能は優れています。**

> Unity-MCPというツールが存在しますが、セキュリティの脆弱性があるため、使用前に必ずセキュリティレビューを行ってください。
{：.prompt-danger}

### MCPサーバーを使用するときの注意

MCP（Model Context Protocol）サーバーを使用する場合：
- 信頼できるソースであるか必ず確認
- セキュリティの脆弱性がないか確認
- チームで使用する場合はホワイトリスト管理を推奨
